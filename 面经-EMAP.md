# 面经：EMAP

## 项目简介（简历可用，1～2 句）

我做的是一个 `Python + FastAPI + LangChain` 的记忆增强 Agent 后端平台，核心能力是文档导入、`BM25 + FAISS + CrossEncoder reranker` 混合检索、带引用问答，以及偏好记忆、权限过滤、安全降级和离线评测闭环。它不是单纯的 RAG Demo，而是面向企业知识库场景的可追溯问答底座。

## 简历 bullet（4～6 条）

- 设计并实现 `FastAPI + Controller / Service / Repository` 分层的 Agent 后端主链，统一承接文档导入、检索、问答与记忆管理能力。
- 落地 `BM25 + FAISS + CrossEncoder reranker` 混合检索方案，将关键词召回、语义召回和重排融合到同一问答链路，提升复杂查询下的召回稳定性（待测）。
- 实现带引用问答与 `output guard` 校验机制，在缺失引用或伪造引用时自动降级为证据式回答，强化结果可追溯性。
- 引入 `input guard + ACL 权限过滤 + 审计日志` 安全闭环，拦截提示词绕过、越权访问和敏感泄漏风险，支撑企业内部知识问答场景。
- 搭建离线评测脚本与 CI 流程，输出 `retrieval_hit_rate`、`context_precision/recall`、`citation_coverage`、安全指标等质量指标，为后续迭代提供回归门禁。

## 面试问题（15 个主问题，按频率从高到低）

### 主题一：系统定位与整体架构

### Q1（频率：高）[通用/后端/AI]
**面试官问题**：你为什么把这个项目定位成“记忆增强 Agent 后端”，而不是普通的 RAG Demo？

**口播版**：我当时做这个项目，不是想停留在“给一段文本、回一句答案”的演示层，而是想把企业知识问答里真正会遇到的问题一起收进来。场景上，普通 RAG Demo 往往只关心检索和生成能不能跑通，但业务里还会追问偏好记忆怎么管理、越权检索怎么拦、回答能不能追溯、质量怎么评估。我的任务就是把这些能力放进一个统一后端主链里，所以我把它拆成文档导入、混合检索、带引用问答、偏好与会话记忆、安全守卫、离线评测几块。最终结果是，这个项目更像一个可持续迭代的知识问答底座，而不是一次性展示模型效果的 Demo，这也是我在面试里会强调的定位差异。

**追问 1**：如果让你一句话讲清这个项目的主链路，你会怎么讲？

**口播版（≥150 字）**：我一般会这样讲：用户先通过导入接口把文档写进系统，系统会做切片、把结构化元数据写到 MySQL、把文本向量写到 FAISS；真正问答时，接口层只负责收请求，核心逻辑在 `IntelligentQA` 里统一编排，先做输入安全检查，再做文档权限过滤，然后走 `BM25 + 向量 + reranker` 的混合检索，把召回到的证据和用户偏好一起交给 LLM 或兜底模板生成答案，最后再经过输出引用校验和审计记录返回给用户。这样讲的好处是，面试官能一下听出我不是只会“调模型”，而是把数据流、控制流和风险控制一起设计过。

**追问 2**：为什么采用 `controller -> service -> repository` 这种分层？

**口播版（≥150 字）**：我采用这套分层，核心考虑是让 HTTP 细节、业务编排和数据访问各自有边界。场景上，这个项目既有导入链，也有问答链、安全链、评测链，如果把逻辑直接堆到 controller 里，后面任何能力扩展都会变得难维护。我的任务是保证主链能演进，所以接口层只负责参数与路由，service 层负责业务顺序和规则编排，repository 层只负责 MySQL、Redis、FAISS 这些底层读写。结果上，这样做以后，像 `input_guard`、ACL、output guard、audit log 这些横切逻辑都能比较自然地插进 service 主链里，测试也能针对单层做隔离验证，工程可维护性会明显更好。

### Q2（频率：高）[后端/AI]
**面试官问题**：你是怎么理解并拆分“三类记忆”的？

**口播版**：我理解的三类记忆，本质上是三种完全不同的生命周期和治理方式。场景上，用户的输出风格偏好、业务知识文档、一次会话里的上下文消息，不能放在同一个抽象里处理，否则删除、追溯、缓存和权限策略都会混乱。我的任务是先把边界打清楚，所以我把偏好记忆作为长期结构化配置，放在 MySQL；把会话记忆作为短时状态，放在 Redis 并带 TTL；知识记忆则通过导入、切片、索引和检索链路来承载。当前仓库已经把偏好记忆和会话记忆落地，知识记忆的检索主链已经成型，但完整的编辑、删除、追溯闭环还可以继续补，这是我会诚实说明的地方。

**追问 1**：偏好记忆具体怎么影响回答？

**口播版（≥150 字）**：我这里没有把偏好记忆做成花哨的“人格系统”，而是先解决最稳定、最好验证的一层。场景上，用户常见的偏好是输出长度、语言习惯、回答风格，这些信息如果每次都靠前端重复传，成本高也不一致。我的做法是把偏好通过 `/v1/memory/preferences` 写进 MySQL，在问答链里先查出当前用户的偏好，然后在 LLM 提示词或兜底回答中显式带上。结果上，即使 LLM 关闭，模板兜底回答也能体现“已应用偏好”，这说明记忆不是摆设，而是真的接进了主链。后续如果继续演进，我会把偏好从简单键值扩展成更细的策略模板。

**追问 2**：为什么不把全部历史会话直接塞进 prompt？

**口播版（≥150 字）**：我没有选择“全量历史直接塞 prompt”，是因为那种方案在 Demo 阶段很省事，但在真实知识问答里副作用很大。场景上，会话一长，token 成本和噪声都会快速上升，而且历史里很多内容其实跟当前问题无关，反而会稀释真正重要的检索证据。我的任务是让系统更可控，所以我把短时会话放在 Redis，用 TTL 和消息数上限控制长度，把知识部分交给检索链处理，而不是把一切都堆给模型。结果是，主链可以保持“证据优先”的结构，模型看到的是有限的会话状态加高相关证据，而不是一大坨历史垃圾文本，这样更稳定也更便于审计。

### Q3（频率：高）[通用/AI]
**面试官问题**：为什么你把“来源引用”和“可追溯”当成硬约束？

**口播版**：我把来源引用当成硬约束，是因为这个项目瞄准的是企业知识问答，而不是闲聊机器人。场景上，用户问的往往是制度、知识库文档、内部流程，这类答案一旦没有证据支撑，风险很高。我的任务不是让模型“看起来像知道”，而是让用户知道答案为什么成立，所以我要求最终输出必须引用当前命中的 `chunk_id`。如果模型没引用、引用错了，系统不会硬返回，而是由 `output_guard` 改成证据式回答。结果上，哪怕回答更保守一些，也能保证用户看到的是可核对的信息，这对知识库场景比“流畅废话”有价值。

**追问 1**：如果模型生成了假的引用，你怎么处理？

**口播版（≥150 字）**：这个场景我专门做了守卫，因为大模型很容易“格式对了但引用是编的”。我的处理顺序是：先拿答案里抽出的 `chunk_id` 和当前检索命中的 `hits` 做集合比对，只要发现引用不在命中列表里，就判定为 `citation_mismatch`，不直接把原答案返回给用户，而是自动降级成“只列证据、不做自由发挥”的保守回答。这样做的任务目标很明确，就是宁可少说，也不让系统用伪造引用包装错误答案。最终结果是，引用校验从“提示词要求”变成了“程序硬校验”，风险边界会清晰很多，而且这块还有对应测试验证，不是我嘴上说说。

**追问 2**：如果完全没有检索到证据，你会怎么回答？

**口播版（≥150 字）**：我这里不会强行让模型编一个像样答案，因为那样最容易把系统带偏。场景上，如果当前文档库里没有相关内容，或者权限过滤后可用文档为空，继续生成只会制造幻觉。我的处理是直接返回“当前还没找到可引用证据”，并提示用户先导入文档或者补充查询范围，同时审计链里也会留下当前请求、风险等级和命中结果。这样做的任务目标，是把“知识缺失”和“模型能力不足”明确暴露出来，而不是用自然语言把问题盖过去。结果上，系统虽然看起来更克制，但用户行为会更可预期，也更适合企业内部场景。

### 主题二：导入链路与存储设计

### Q4（频率：高）[后端/AI]
**面试官问题**：文档导入链路你是怎么设计的？

**口播版**：我把导入链路设计成“解析、切片、替换元数据、重建向量”的顺序。场景上，知识库最常见的问题不是第一次导入，而是后续更新同一份文档时怎么避免旧内容残留。我的任务是保证导入后的检索结果能和最新文档一致，所以在 `Pipeline.ingest` 里，我先根据 `document_id` 查已有 chunk，如果有旧 chunk 就先从向量索引里删掉，再用新的切片结果替换 MySQL 里的元数据，最后把新 chunk 重新向量化写进 FAISS。结果上，主链至少把“更新文档后还召回旧证据”这个典型脏数据问题纳入了控制范围，这比单纯 append 更接近真实工程要求。

**追问 1**：为什么支持 `recursive`、`markdown`、`sentence` 三种切片策略？

**口播版（≥150 字）**：我支持多切片策略，核心不是为了“做得花”，而是因为文档类型不同，最佳边界也不同。场景上，Markdown 文档有明确标题层级，直接按自然段切会打散结构；一些说明文又更适合按句切，避免一个 chunk 混进太多主题。我的任务是先提供最小但有效的可选方案，所以用 `RecursiveCharacterTextSplitter` 做统一入口，再根据策略调整分隔符优先级。结果上，系统至少具备按文档类型优化切片的能力，后续可以通过离线评测比较不同策略下的命中率与引用完整性，而不是把切片当成一个永远不动的黑盒。

**追问 2**：更新文档时，你怎么保证向量索引不脏？

**口播版（≥150 字）**：我对这个问题的处理重点是“先删旧、再写新”，而不是无脑覆盖。场景上，如果只更新 MySQL 的 chunk 元数据，不删 FAISS 里的旧向量，就会出现主表已经是新内容，但检索时还能命中旧 chunk 的情况。我的做法是，在 `Pipeline.ingest` 里先拿到同一 `document_id` 的旧 chunk 列表，调用 `vector_repository.remove_document_chunks` 删除映射和向量，再执行 `replace_chunks` 写入新元数据，最后把新 chunk 重新做 embedding 并持久化索引。结果上，至少在当前单机 FAISS 方案下，主链考虑到了元数据和向量索引的一致性，而不是只顾“导入成功”这个表象。

### Q5（频率：高）[后端]
**面试官问题**：为什么把 MySQL、Redis、FAISS 拆开用，而不是统一放一处？

**口播版**：我拆开用的原因，是三类数据的访问模式和治理方式完全不同。场景上，文档元数据、用户偏好这类结构化信息适合 MySQL，短时会话状态适合 Redis，语义近邻检索适合向量索引；如果强行放一处，要么查询能力受限，要么一致性和成本都很难兼顾。我的任务是先把职责边界理顺，所以用了 MySQL 管结构化数据，Redis 管短时消息，FAISS 管语义向量。最终效果是，系统在 MVP 阶段就具备比较清楚的分工，后续也方便把 FAISS 替换成独立向量库，而不用把整条链全推翻重来。

**追问 1**：`metadata_repository` 为什么要自己做 schema 初始化和重试？

**口播版（≥150 字）**：我在 `metadata_repository` 里自己做 schema 初始化和简单重试，主要是因为这个仓库当前目标是“最小可运行且能联调”，而不是先引入一整套重型迁移框架。场景上，项目刚起步时，如果连表都要手动准备，大家本地跑起来会很痛苦；同时 MySQL 连通性在开发环境里也经常有抖动。我的任务是让基础链路尽快稳定，所以仓库初始化时自动确保 `documents` 和 `chunks` 表存在，执行 SQL 时用有限次重试和退避。结果上，本地和联调环境更容易跑通，但我也会坦诚说，后续生产化最好升级成显式迁移管理，而不是长期依赖运行时建表。

**追问 2**：FAISS 本地持久化方案的局限是什么？

**口播版（≥150 字）**：FAISS 本地持久化是我在 MVP 阶段做的权衡，它的优点是快、轻、依赖少，但局限也很明确。场景上，当文档规模继续增长、部署从单机走向多实例、或者需要更复杂的过滤与在线维护时，本地索引文件就会成为瓶颈。我的任务是在当前阶段先把语义召回链打通，所以我用 `IndexIDMap2` 做向量 ID 映射，再把 mapping 单独持久化成 JSON。结果上，这套方案足够支持当前原型，但我在 README 里也明确把“替换独立向量库”作为后续演进方向。面试时我会把这说成“有意识的阶段性取舍”，而不是假装它已经是生产级最终形态。

### Q6（频率：中高）[后端/AI]
**面试官问题**：本地文件导入能力是怎么做的？

**口播版**：我做本地文件导入的目标，是让项目不仅能吃接口里的纯文本，也能吃真实知识库会遇到的 `.md`、`.txt`、`.pdf`。场景上，很多团队的知识原始材料并不规整，如果只能粘贴字符串，系统落地价值很有限。我的做法是保留自己对业务链的控制，但借助 `LangChain + UnstructuredLoader` 做文件解析，把解析后的文本统一交给现有切片和入库链。结果上，项目在工程边界上比较清楚：LangChain 负责帮助我接入解析器，不替代我自己的记忆、检索和权限模型，这样既复用了生态，又没有把业务核心交出去。

**追问 1**：为什么用了 LangChain，但又强调“不让它接管业务核心”？

**口播版（≥150 字）**：我对 LangChain 的使用原则很明确，就是把它当成工作流编排和生态接入工具，而不是业务真相来源。场景上，很多项目一上来把检索、记忆、提示词、Agent 行为都塞进框架抽象，短期开发很快，但后期一旦要加权限、审计、评测、引用约束，就很容易被框架默认行为绑住。我的任务是做一个可控的后端底座，所以我只在文件解析、文本分割这类适合借力的地方用 LangChain，核心的问答主链、数据模型、安全与评测还是自己写。结果上，项目保留了工程可解释性，也更适合面试里讲清楚每一步为什么存在。

**追问 2**：如果后续要支持扫描件 OCR，你会怎么扩展？

**口播版（≥150 字）**：如果后续要支持扫描件 OCR，我不会直接把 OCR 逻辑硬塞进当前同步导入接口，而是会把它当成一个异步解析能力来扩展。场景上，扫描件解析耗时高、错误率也更复杂，如果仍然走同步链路，接口体验和失败恢复都会比较差。我的计划是把文件上传、解析任务、OCR 结果回写拆成几步，必要时加 MQ 或后台任务，把原始文件、解析状态、失败原因都记录下来。结果上，导入链就能从当前的本地解析原型，平滑演进成更接近生产的文档处理流水线，同时不会污染现在已经比较稳定的同步 ingest 主链。

### 主题三：检索、生成与引用控制

### Q7（频率：高）[AI/后端]
**面试官问题**：你们的混合检索具体是怎么落地的？

**口播版**：我这里的混合检索不是“文档里写了一句关键词加向量”，而是把三段逻辑明确拆开并落到代码里。场景上，单一路径召回往往不稳定，关键词能抓显式术语，向量能补语义近邻，但两者都不一定能直接给出最优排序。我的任务是把召回和排序分开，所以先让 BM25 和向量各自召回，再把候选集合并做一轮预排序，然后把候选池交给 CrossEncoder reranker 重新打分，最后再按权重做融合。结果上，主链既能解释为什么某个 chunk 被选中，也能通过配置调整各路权重，这种可解释与可调优性是我认为比较像工程项目的点。

**追问 1**：三路权重是怎么配的？你怎么讲这个取舍？

**口播版（≥150 字）**：我没有把权重写死在代码里，而是全部放进配置，原因是不同业务语料对关键词和语义检索的依赖程度并不一样。场景上，制度文档、术语文档更吃关键词，开放问答和长文本摘要更吃语义相似度，而 reranker 更像最后一层精排。我的任务不是一次把最优参数猜出来，而是把调参入口留出来，所以仓库里暴露了 `HYBRID_BM25_WEIGHT`、`HYBRID_VECTOR_WEIGHT`、`HYBRID_RERANKER_WEIGHT` 等配置项。结果上，我可以先用默认权重跑通，再结合离线评测逐步校准，而不是一开始就把“经验值”包装成不可质疑的最优解。

**追问 2**：为什么不是直接把所有候选都丢给 reranker？

**口播版（≥150 字）**：我没有把全量 chunk 全丢给 reranker，主要是为了控制成本和延迟。场景上，CrossEncoder 的精度通常比双塔召回高，但计算代价也大，如果候选池不先收窄，文档量一上来就会拖垮请求延迟。我的任务是把精排用在最有价值的位置，所以先用 BM25 和向量召回做一轮融合预排序，再截断到 `candidate_pool_size`，只让最有可能相关的一批 chunk 进入 rerank。结果上，系统在效果和成本之间形成了一个明确的折中点，也为后续做超时降级、异步重排或独立检索服务留出了空间。

### Q8（频率：高）[AI]
**面试官问题**：BM25、FAISS、CrossEncoder reranker 三者分别解决什么问题？

**口播版**：我会把这三者理解成检索链上三个职责完全不同的角色。场景上，BM25 更擅长处理显式关键词、专有名词和精确术语，FAISS 负责语义相似检索，能在关键词不完全匹配时补充候选，而 CrossEncoder reranker 更像最终裁判，它拿 query 和候选 chunk 成对判断相关性。我的任务是让这三者形成互补而不是互斥，所以我把 BM25 和 FAISS 作为召回层，把 reranker 放到精排层。结果上，项目能同时兼顾“查得着”和“排得对”两件事，这比只做向量相似度更容易讲出技术深度。

**追问 1**：如果文档规模继续变大，当前方案的瓶颈会在哪里？

**口播版（≥150 字）**：我觉得当前阶段最明显的瓶颈，不在 FAISS 本身，而在 BM25 的实现方式和整体单机场景假设。场景上，仓库里的 BM25 目前还是把 MySQL 里的 chunk 拉出来，在进程内做计算，这对 MVP 没问题，但数据量一大，扫描、内存和请求时延都会变差。我的任务是在原型阶段先把召回逻辑证明出来，所以接受了这个取舍。结果上，下一步演进我会优先把 BM25 换成独立全文检索能力，比如 ES/OpenSearch/Tantivy 一类方案，再把检索和问答服务解耦，否则规模一上去，当前进程内方案会先扛不住。

**追问 2**：如果 reranker 模型不可用，你准备怎么兜底？

**口播版（≥150 字）**：我会把 reranker 不可用视为“效果下降”，而不是“整个系统不可用”。场景上，模型首次预热失败、下载失败或者推理超时都可能出现，如果这时系统直接报错，体验会非常差。我的任务是保证主链有降级路径，所以即使 reranker 不可用，BM25 和向量召回仍然可以先组成一个可用的候选排序，系统继续返回带引用答案，只是相关性可能下降。结果上，用户至少拿到的是有证据的答案，而不是空响应。后续如果继续工程化，我会补充更显式的健康检查、预热脚本和超时监控，把 reranker 从“可用就加分”变成“可观测、可降级”的组件。

### Q9（频率：高）[AI/后端]
**面试官问题**：问答链路里，用户偏好和检索证据是怎么一起作用的？

**口播版**：我的原则是“证据决定事实，偏好只影响表达”。场景上，如果把偏好放得太重，很容易把用户习惯变成事实判断的一部分，这是知识问答里不能接受的。我的任务是把两者层次分清楚，所以先通过混合检索拿到证据，再查询该用户的偏好，把偏好作为输出风格的附加信息注入提示词或兜底回答，但不允许它影响证据选取。结果上，系统既能保持答案事实来源的稳定性，又能体现出记忆能力，而不是把偏好做成一个会污染检索判断的黑盒特性。

**追问 1**：如果 LLM 没开，系统怎么保证还能工作？

**口播版（≥150 字）**：我没有把系统可用性绑死在外部 LLM 上，因为很多联调场景、内网场景或者网关不稳定时，模型能力不一定随时在线。我的任务是让问答链在没有 LLM 的时候仍然可以“可信地退化”，所以做了模板化兜底回答：先把检索证据列出来，再结合偏好做一个保守结论，至少保证用户能看到命中了哪些 chunk。结果上，即使 `LLM_ENABLED=false`，系统依然不是瘫痪状态，而是从“生成式回答”退回“证据式回答”。这种设计对工程项目很重要，因为它把核心能力定义成“证据检索 + 可追溯输出”，而不是“必须依赖外部模型在线生成”。

**追问 2**：为什么要用结构化解析 `final_output`，而不是直接拿模型返回文本？

**口播版（≥150 字）**：我让模型按结构化格式返回，核心是为了控制结果边界。场景上，大模型很容易在回答前后加解释、思考痕迹、Markdown 包裹甚至奇怪噪声，如果直接把原文本透传给用户，输出质量很不稳定。我的任务是让主链只接受“最终可展示结果”，所以我用 `PydanticOutputParser` 约束模型只返回 `final_output`，并且在解析失败时做二次清洗和兜底。结果上，模型输出从“任意字符串”收敛成“我能校验和后处理的结构”，这让后面的引用校验、安全审查和审计落盘都更好做，也更符合工程系统而不是聊天脚本的思路。

### Q10（频率：高）[AI]
**面试官问题**：你怎么保证回答里的引用是真的、有效的？

**口播版**：我处理这个问题，不靠“相信模型会听话”，而是靠程序校验。场景上，只要系统允许自由生成，就必须假设模型可能少引、错引、乱引，所以我把引用合法性校验放在输出阶段。具体做法是：先从答案文本里抽取 `chunk_id`，再和当前请求实际命中的 `SourceItem` 列表做比对，如果答案有非法引用，或者有命中证据但答案完全没引用，就直接降级成证据式回答。结果上，引用从“文案要求”变成了“结果门禁”，这使得系统返回给用户的内容至少是可核对的，也为后续做 `citation_validity_rate` 评测提供了基础。

**追问 1**：`output guard` 在这里具体做了什么？

**口播版（≥150 字）**：`output guard` 在我这里不是一个抽象概念，它承担了三个很具体的职责。第一，校验答案是否为空，避免模型给出不可展示的无效结果；第二，检查答案里是否包含敏感泄漏内容，比如明显的凭证、密码类信息；第三，也是最关键的，校验答案引用是否真实命中当前召回证据。如果没有引用、引用和命中不匹配，或者高风险场景下回答不够保守，它都会把最终输出改成 grounded 的证据式回答。结果上，主链不是“模型一说完就结束”，而是还有一个工程侧的结果闸门，这个设计能明显提升系统的可信度。

**追问 2**：你怎么理解 `citation_coverage_rate` 这个指标？

**口播版（≥150 字）**：我把 `citation_coverage_rate` 理解成“答案内容有多少比例真正能被当前上下文支撑”。场景上，光看有没有引用是不够的，因为模型可能只在最后挂一个 `chunk_id`，中间大段内容仍然是自由发挥。我的任务是让评测更接近真实风险，所以在离线评测里会把答案 token 和检索上下文 token 做覆盖度计算，看看最终答案有多少内容能在证据里找到来源。结果上，这个指标不能完全等价于真实性，但它能帮助我发现“引用形式正确、内容却漂了”的情况。面试时我也会说明，它是一个工程近似指标，不是万能真理裁判。

### 主题四：安全、权限与审计

### Q11（频率：高）[安全/AI]
**面试官问题**：输入安全和高风险降级这块，你是怎么设计的？

**口播版**：我把输入安全分成两层看，一层是直接拦截明显不合规请求，另一层是对高风险问题做保守降级。场景上，知识问答系统很容易碰到 prompt injection、敏感信息探测、危险请求这些问题，如果所有问题都走同一条自由生成链路，风险很高。我的任务是先把高概率风险挡在前面，所以 `input_guard` 里针对绕过系统提示、泄露密钥、危险行为等做规则匹配；如果是合规、权限、审计、隐私这类高敏问题，即便允许继续问答，也默认走 grounded-only 模式。结果上，系统把“能不能答”和“应该答到什么程度”分开了，这比简单的 allow / deny 更贴近真实场景。

**追问 1**：prompt injection 你具体怎么拦？

**口播版（≥150 字）**：我对 prompt injection 的处理思路是先做明确模式拦截，再做链路级隔离。场景上，很多攻击会直接要求“忽略系统提示”“显示 hidden prompt”“绕过 guard”，这些语义在知识问答场景里没有正当业务价值，所以我先在 `input_guard` 里用规则做第一层过滤，只要命中就直接返回拒答，不进入检索和生成。我的任务不是把所有高级攻击一次解决，而是先把最常见、最便宜的攻击面关掉。结果上，主链至少能拦住明显绕过意图，而且从测试层面也验证了这类请求会被 block，而不是靠提示词碰碰运气。

**追问 2**：为什么高风险问题不直接拒答，而是改成 grounded-only？

**口播版（≥150 字）**：我没有把所有高风险问题一刀切拒绝，是因为像合规、权限、审计、隐私治理这类问题，本身可能就是合法业务问题。场景上，用户可能真的是安全人员、审计人员，完全拒答会把系统做得过于死板。我的任务是平衡安全性和可用性，所以对这类问题我不让模型自由发挥，而是只允许它整理当前检索到的证据，明确告诉用户“我现在给的是保守结论”。结果上，系统既没有因为追求安全把价值做没，也没有因为追求聪明放开模型自由发挥。我认为这是企业知识库场景里更合理的中间态。

### Q12（频率：中高）[安全/后端]
**面试官问题**：权限过滤是怎么做的？

**口播版**：我把权限过滤做在检索前，而不是做完检索再决定回不回答。场景上，如果系统先把私有文档召回出来，再在回答阶段说“你无权访问”，其实已经产生了泄漏风险。我的任务是把权限约束前置，所以我设计了 `AccessController`，用用户 ID 对应一组允许的 `document_id` 前缀，再和当前请求传入的前缀做交集，只有最终有效的前缀范围内的文档才能参与召回。结果上，越权控制进入了检索域，而不是停留在接口文案层，这对企业内部资料问答非常关键。

**追问 1**：基于 `doc_id_prefix` 的 ACL 有什么边界？

**口播版（≥150 字）**：我很清楚这套 ACL 现在还是一个轻量实现，它解决的是“最小可用的文档范围控制”，不是完整权限平台。场景上，前缀规则适合按资料集、团队、项目做粗粒度隔离，比如 `doc-team-a`、`doc-private` 这种命名空间，但它不适合细到单 chunk、单字段或复杂角色继承。我的任务是在 MVP 阶段先证明“权限必须进入检索主链”这件事，所以先选了可实现、可测试的前缀交集模型。结果上，它足以支撑当前原型，也方便我在面试里讲清楚边界；后续如果继续做，我会把它升级到 chunk 级 metadata ACL 或更完整的 RBAC/ABAC。

**追问 2**：未来如果要升级权限模型，你会怎么做？

**口播版（≥150 字）**：如果继续演进，我会优先做两件事。第一，把权限从“请求传前缀 + 用户静态规则”升级成更细粒度的 metadata ACL，让文档或 chunk 自带租户、部门、敏感级别标签；第二，把权限决策和配置管理独立出来，支持后台维护和审计。场景上，企业资料通常不是简单的一层目录结构，单靠前缀很快会不够用。我的任务会从“拦住明显越权”升级成“权限可配置、可审计、可回放”。结果上，权限控制才能真正从 Demo 级别走向平台级别，而且与现有 `AccessController` 的设计并不冲突，只是把现在的粗粒度交集规则往更细的策略引擎方向扩展。

### Q13（频率：中高）[安全/工程]
**面试官问题**：审计日志和隐私脱敏这块，你做了什么？

**口播版**：我做审计日志，不是为了“看起来像有安全意识”，而是因为问答系统一旦出现越权、拒答、错误引用，必须能回放上下文。场景上，很多系统出了问题之后只剩一条应用日志，根本不知道用户问了什么、命中了哪些文档、为什么触发降级。我的任务是把这些关键过程信息留下来，所以我在主链里记录用户 ID、会话 ID、风险等级、请求前缀、实际生效前缀、命中 chunk、是否 blocked、是否 degraded 等信息，并在写日志前对 query 和 answer 做脱敏处理。结果上，系统至少具备了基础的事后分析能力，而不是出了问题只能靠猜。

**追问 1**：你会把哪些字段记进审计日志？

**口播版（≥150 字）**：我会优先记录那些既能解释系统行为、又不会额外暴露原始敏感数据的字段。具体来说，最重要的是用户标识、会话标识、标准化后的 query、风险等级、输入 guard 标签、请求前缀与最终生效前缀、是否被 block、是否被降级、命中的 chunk ID 列表，以及最终返回给用户的答案摘要。场景上，这些信息足够支撑事后追查“为什么答成这样”“为什么这次越权了没有”之类的问题。我的任务不是把所有原文一股脑存下来，而是在可回放与隐私保护之间做平衡，所以我在落盘前会做敏感信息脱敏处理，避免审计日志自己变成风险源。

**追问 2**：本地 JSONL 审计日志的局限是什么？

**口播版（≥150 字）**：本地 JSONL 审计日志适合开发和原型验证，但我不会把它包装成生产终态。场景上，单机文件写入在并发、检索、归档、告警和权限管理上都比较弱，一旦部署成多实例，日志就会分散在不同机器上，排查成本很高。我的任务是在当前阶段先把审计字段和调用时机做对，所以用 JSONL 作为最小实现。结果上，开发阶段已经能回放问答链路，但后续如果走生产，我会把它接到集中式日志或可观测平台，增加检索、告警、聚合与保留策略，这样审计能力才算真正闭环。

### 主题五：评测闭环与工程化

### Q14（频率：中高）[工程/AI]
**面试官问题**：你是怎么做离线评测和质量门禁的？

**口播版**：我不希望这个项目只能靠“试几次感觉还行”来评估质量，所以专门补了一条离线评测链。场景上，检索、引用、安全这几块一旦迭代，很容易出现隐性回归，人工点接口很难稳定发现问题。我的任务是把质量变成可重复的结果，所以我做了 `EvaluationRunner` 和 `scripts/evaluate_rag.py`，支持加载样本集，跑问答链，统计 `retrieval_hit_rate`、`context_precision/recall`、`citation_coverage_rate`，以及拒答匹配率、引用有效率、禁区 chunk 泄漏率等安全指标。结果上，系统具备了最小的离线质量闭环，后续可以用阈值门禁把“感觉”变成“标准”。

**追问 1**：什么样的样本你会放进 `eval` 数据集？

**口播版（≥150 字）**：我会把样本分成三类来构造。第一类是标准知识问答样本，重点验证能不能命中正确 chunk，并返回有引用的答案；第二类是边界样本，比如同义改写、长问题、术语密集问题，用来验证混合检索的稳定性；第三类是安全与权限样本，比如 prompt injection、需要拒答的问题、限制文档前缀的问答、禁止命中文档的样本。场景上，如果评测集只有“正常问题”，你永远不知道系统在异常路径上会不会翻车。我的任务是让评测覆盖主链和风险链，所以会同时放业务正确性样本和安全约束样本，这样评测结果才有实战意义。

**追问 2**：怎么把这些评测接进 CI 做门禁？

**口播版（≥150 字）**：我当前已经把 `pytest` 接进 CI，下一步很自然就是把离线评测也作为补充门禁。场景上，单元测试更多验证逻辑正确性，但像检索命中率、引用覆盖率这种质量指标，需要跑评测脚本才能得到。我的任务会是给评测脚本增加最小阈值参数，比如 `--min-retrieval-hit-rate`、`--min-context-precision`、`--min-citation-coverage`，一旦结果低于阈值就返回非零退出码，让 CI 直接失败。结果上，每次改检索权重、切片策略或安全逻辑时，都能第一时间看到质量是否退化，而不是等上线后靠用户反馈倒逼修复。

### Q15（频率：中）[通用]
**面试官问题**：你觉得这个项目最有价值的亮点和当前最大的短板分别是什么？

**口播版**：我觉得这个项目最有价值的亮点，不是“用了哪些模型名词”，而是把企业知识问答里几个容易被忽略但很关键的能力收进了同一条主链，包括混合检索、引用校验、安全降级、权限过滤和离线评测。场景上，很多项目只做到能答，没做到可控、可追溯、可评估。我的任务是先把这些地基铺起来，所以现在仓库已经能比较完整地展示一个工程化 RAG 后端的骨架。最大的短板也很明确：BM25 还是进程内计算，向量索引还是本地 FAISS，知识记忆的完整治理还没全部做完。结果上，这个项目适合被讲成“方向正确、骨架扎实、仍在继续工程化”的主项目，而不是包装成已经完全生产成熟。

**追问 1**：如果再给你一周时间，你会优先做什么？
