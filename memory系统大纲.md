# Memory 系统大纲（可执行版 v1）

更新时间：2026-05-26

## 1. 目标与约束

### 1.1 目标

采用“ChatGPT 四层运行时结构 + OpenViking 记忆编排思想”落地，兼顾临时交互和长期沉淀，控制 Token 成本与记忆膨胀。

### 1.2 强约束（与仓库规则对齐）

1. 记忆数据模型固定三类：偏好记忆、知识记忆、会话记忆。
2. 检索必须混合策略：关键词 + 向量 + 重排。
3. 回答必须保留来源引用。
4. 记忆必须支持可编辑、可删除、可追溯。
5. 禁止把全部历史会话直接塞入上下文。

### 1.3 默认方案与备选

1. 默认方案：四层仅作为运行时组织方式，底层持久化仍映射到三类记忆模型。
2. 备选方案：完全按三类建模，不显式定义四层；实现更快，但可解释性和治理精度较弱。

## 2. 四层运行时到三类记忆映射

| 四层运行时 | 作用 | 对应三类记忆 | 存储 |
|---|---|---|---|
| 会话元数据层 | 当前会话设备/渠道/时段等轻量上下文 | 会话记忆 | Redis（短 TTL） |
| 当前会话滑动窗口 | 保存最近 N 轮原始对话 | 会话记忆 | Redis List |
| 近期摘要层 | 每 N 轮生成摘要用于跨会话衔接 | 会话记忆（摘要子层） | MySQL + 向量库 |
| 长期记忆层 | 稳定偏好与长期事实 | 偏好记忆 + 知识记忆 | MySQL + 向量库 |

说明：OpenViking 用于记忆提取/提交/编排思路，不替代本项目核心业务模型与持久化设计。

## 3. 写入与流转规则

### 3.1 会话记忆

1. 用户消息与助手消息写入 Redis 滑动窗口。
2. 窗口上限由 token 预算和消息数双阈值控制，超限 FIFO。
3. 会话结束或 TTL 到期自动清理。

### 3.2 近期摘要

1. 每 8 轮生成一次摘要（可配置 `summary_interval_rounds`）。
2. 摘要仅保留用户目标、偏好变化、未完成事项，不保留冗余闲聊。
3. 摘要参与后续检索，但权重低于长期记忆。

### 3.3 长期记忆沉淀

1. 仅两类内容可入长期：显式记忆指令、稳定性判定通过的信息。
2. 稳定性判定默认阈值 `stability_score >= 0.85`（可配置）。
3. 写入前去重：语义相似度阈值默认 `0.90`，超过阈值走覆盖更新并保留版本。

### 3.4 衰减与确认

1. 长期记忆超过 180 天未命中检索，状态变更为 `pending_confirm`。
2. 每月最多一次确认提问；连续 2 次否定则软删除。
3. 删除后同步清理向量索引与缓存映射，保留审计记录。

## 4. 数据治理与安全

1. 入库前做敏感信息过滤（PII mask/redact），默认手机号/证件号/密钥不入长期记忆。
2. 所有长期记忆写操作记录版本：`version + changed_by + changed_at + change_reason`。
3. 提供显式接口：新增、更新、删除、回滚、查询历史。
4. 所有删除均采用软删除 + 异步物理清理，避免检索与存储不一致。

## 5. Token 与膨胀控制策略

1. 会话启动时冻结长期记忆快照，本会话中新增记忆默认下个会话生效。
2. Prompt 注入顺序：当前问题 > 会话窗口 > 近期摘要 > 长期记忆（按预算截断）。
3. 上下文预算按类型分桶：`window/summary/long_term`，超过预算先裁剪低优先级内容。
4. 目标是降低上下文冗余，实际节省比例以 A/B 评估结果为准，不写死固定百分比。

## 6. 分阶段落地方案

### 阶段 A（Schema 与接口，1-2 天）

1. 定义三类 memory 表结构与索引策略。
2. 增加 `delete/update/history` API。
3. 验收：可编辑、可删除、可追溯闭环跑通。

### 阶段 B（四层运行时，2-3 天）

1. 接入滑动窗口、摘要任务、会话快照冻结。
2. 实现记忆写入规则与去重规则。
3. 验收：长对话不爆 context，跨会话能命中近期摘要。

### 阶段 C（长期记忆治理，2-4 天）

1. 偏好/知识双轨沉淀与版本化。
2. 衰减确认与软删除清理链路。
3. 验收：删除后检索不可见，历史可审计可回滚。

### 阶段 D（评估与上线门禁，1-2 天）

1. 增加最小评测：记忆命中率、去重覆盖率、引用完整率、Token 变化率。
2. 设定阈值门禁，不达标禁止发布。
3. 验收：离线评测稳定通过，线上灰度指标无明显回退。

## 7. 最小可验证测试（必须）

1. 检索链路：关键词 + 向量 + 重排均参与，且能稳定返回引用来源。
2. 删除一致性：删除后 MySQL、向量库、Redis 缓存结果一致。
3. 摘要有效性：跨会话提问可命中最近摘要信息。
4. 安全性：含敏感信息样本不会写入长期记忆。

## 8. 当前边界

1. 当前仓库已具备：偏好记忆（MySQL）+ 会话窗口（Redis）基础能力。
2. 当前仓库未具备：知识记忆层、摘要层、版本回滚、衰减确认全链路。
3. OpenViking 作为编排参考可接入；若接入成本过高，可先按同等规则原生实现。
